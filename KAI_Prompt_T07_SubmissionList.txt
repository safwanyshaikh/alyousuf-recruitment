OBJECTIVE
─────────
Build one screen: Requirement Submission List.

Recruiter workflow this screen must complete:

  Create Requirement → Match Candidate → Assign Candidate
  → Open Submission List → Update Recruiter Remark
  → Update Client Remark → Update Status → Save

No PDF. No Excel. No email. No export. Build only this list.

PART A — ASSIGN BUTTON on Matching Screen
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

On the Matching screen (where STRONG / GOOD / POSSIBLE candidates show),
each candidate row/card must have an [+ Assign] button.

On click:
  - Confirm dialog: "Assign [Name] to [Requirement Name]?"
  - POST to GAS:

    fetch(GAS_URL, {
      method: 'POST',
      redirect: 'follow',
      headers: { 'Content-Type': 'text/plain;charset=utf-8' },
      body: JSON.stringify({
        action: 'addSlot',
        token: sessionToken,
        reqId: requirement.reqId,
        kaiNo: candidate.kaiNo,
        rowIndex: candidate.rowIndex,
        candidateName: candidate.name,
        trade: candidate.trade,
        sourceOwner: candidate.gccMobility || '',
        addedBy: currentUser.email || 'recruiter',
        batchNo: '1'
      })
    })

  - Success → button becomes "✓ Assigned" (disabled, grey)
  - Error "already added" → button becomes "Already Assigned" (disabled)
  - Other error → toast, keep button active


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PART B — ENTRY POINT on Requirements Screen
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

On each requirement row, add action: [View Submission List]
Clicking navigates to the Submission List screen passing reqId.

Show a badge on each requirement row: "X assigned"
Count = slots where status ≠ REJECTED for that reqId.
(Fetch this from the existing requirements response — the field is
already returned as matchCounts or fetch separately if not available.)

PART C — SUBMISSION LIST SCREEN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

HEADER
──────
  ← Back to Requirements

  [Client Name]  ·  [Project / Trade]          [4 counter chips]
  [Deploy Country]  ·  Qty: [N]

Counter chips (recompute on every status change):
  Assigned [n]  •  Shortlisted [n]  •  Submitted [n]  •  Selected [n]

  Assigned    = slotStatus = ADDED
  Shortlisted = slotStatus = SHORTLISTED
  Submitted   = slotStatus = SUBMITTED
  Selected    = slotStatus = SELECTED
  (REJECTED hidden by default)


DATA FETCH
──────────
Step 1 — fetch slots:
  GET {GAS_URL}?action=slots&reqId={reqId}&token={token}

  Response: { ok:true, slots:[...], count:N }

  Each slot:
    slotId, reqId, kaiNo, rowIndex, candidateName, trade,
    addedBy, addedAt, slotStatus, notes, updatedAt,
    batchNo, srNo, recruiterRemark, clientRemark, submissionDate

Step 2 — for each slot, fetch candidate details in parallel:
  GET {GAS_URL}?action=candidate&rowIndex={slot.rowIndex}&token={token}

  Merge into slot row: nationality, age, experience, gulfExp,
  education, positionApplied, mobile, email, passportNo,
  currentLocation, kaiSnippet, score

  If candidate fetch fails → show slot data only (name, trade from slot).
  Do NOT fail the whole list.


GRID COLUMNS (in order)
────────────────────────
  Batch  |  Sr  |  Status  |  Name / Nationality  |  AYE Ref  |
  Passport No  |  Position Applied  |  Education  |
  Total Exp  |  Gulf Exp  |  Location  |
  AYE Assessment  |  Recruiter Remark  |  Client Remark  |
  Submission Date  |  Last Updated  |  Actions

Column details:

  Batch:
    slot.batchNo (default "1")

  Sr:
    slot.srNo (auto-assigned by GAS on addSlot)

  Status:
    Coloured badge:
      ADDED        → grey    "Added"
      SHORTLISTED  → blue    "Shortlisted"
      SUBMITTED    → amber   "Submitted"
      SELECTED     → green   "Selected"
      REJECTED     → red     "Removed"  (only if Show Removed toggle is ON)

  Name / Nationality:
    Line 1 bold:  slot.candidateName
    Line 2 muted: nationality | age+"y"  (skip blanks, no dashes)

  AYE Ref:
    slot.kaiNo  (show as-is, e.g. AYE-KAI-2026-005933)

  Passport No:
    candidate.passportNo (from candidate fetch)
    Show "—" if blank

  Position Applied:
    candidate.positionApplied
    Show "—" if blank

  Education:
    candidate.educationLevel + " " + candidate.educationSubject
    Truncate to 40 chars

  Total Exp:
    candidate.experience + "y"

  Gulf Exp:
    candidate.gulfExp (raw text, truncate to 30 chars)
    Show "—" if blank

  Location:
    candidate.currentLocation
    Show "—" if blank

  AYE Assessment:
    candidate.kaiSnippet (first 100 chars)
    Show "—" if blank

  Recruiter Remark:
    Editable inline text field (see edit spec below)
    Shows slot.recruiterRemark
    Placeholder: "Add remark..."

  Client Remark:
    Editable inline text field (see edit spec below)
    Shows slot.clientRemark
    Placeholder: "Add client remark..."

  Submission Date:
    slot.submissionDate
    Set automatically by GAS when status changes to SUBMITTED
    Show "—" if blank

  Last Updated:
    slot.updatedAt formatted as "02 Jun 2026 14:30"

  Actions:
    [👁 Open]  [↕ Status]  [✕ Remove]


INLINE REMARK EDITING
─────────────────────
Recruiter Remark and Client Remark fields are editable inline in the table.

Behaviour:
  - Click the field → becomes an input/textarea
  - Type remark
  - Press Enter or click away → auto-save

  POST on save:
    fetch(GAS_URL, {
      method: 'POST',
      redirect: 'follow',
      headers: { 'Content-Type': 'text/plain;charset=utf-8' },
      body: JSON.stringify({
        action: 'updateSlot',
        token: sessionToken,
        slotId: slot.slotId,
        recruiterRemark: newValue,   // for recruiter remark field
        // OR
        clientRemark: newValue,      // for client remark field
        actor: currentUser.email
      })
    })

  On success: update the cell value in state, show brief "Saved ✓" indicator
  On error: revert to previous value, show error toast


CHANGE STATUS
─────────────
Clicking ↕ opens an inline dropdown:
  ○ Added
  ○ Shortlisted
  ○ Submitted
  ○ Selected
  ○ Rejected

Current status pre-selected.

POST on select:
  body: { action:'updateSlot', token, slotId, newStatus:'SHORTLISTED', actor }

On success:
  - Update badge immediately
  - Recompute counter chips
On error:
  - Revert badge
  - Show error toast


REMOVE CANDIDATE
────────────────
Clicking ✕:
  Confirm dialog: "Remove [Name] from this requirement?"
  [Cancel]  [Remove]

On confirm:
  POST: { action:'updateSlot', token, slotId, newStatus:'REJECTED', actor }

On success:
  - Hide row (or show as red "Removed" if Show Removed toggle is ON)
  - Recompute counters
  - Toast: "Candidate removed"

"Show Removed" toggle — top right of table, default OFF.
When ON: show REJECTED rows with red badge. No action buttons on removed rows.


OPEN CANDIDATE
──────────────
Clicking 👁 opens the existing 5-tab candidate drawer as an overlay.
Pass candidate.rowIndex to the drawer.


EMPTY STATE
───────────
  [icon]
  No candidates assigned yet.
  [+ Add Candidates →]

Button navigates to Matching screen for this requirement.


LOADING
───────
Show skeleton rows while slots load.
Show per-row spinner while candidate details load.
Table appears immediately with slot data; candidate columns fill in as fetched.

Endpoints this screen uses:
  GET  action=slots&reqId=         → list of slots for a requirement
  GET  action=candidate&rowIndex=  → full candidate profile (per slot)
  POST action=addSlot              → assign candidate
  POST action=updateSlot           → change status OR update remarks OR both

updateSlot accepts any combination:
  { action:'updateSlot', token, slotId, actor,
    newStatus?,        — optional
    recruiterRemark?,  — optional
    clientRemark?,     — optional
  }

Slot statuses (exact uppercase strings):
  ADDED  SHORTLISTED  SUBMITTED  SELECTED  REJECTED  INTERVIEWED  DEPLOYED
